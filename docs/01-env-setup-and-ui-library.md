# 01차: 개발 환경 구축 및 UI 라이브러리 준비/셋팅

> 상세화 전 검토 체크포인트  
> - 기본 개발 스택(Node, 패키지 매니저, Next.js 버전) 기준이 확정되었는가?  
> - UI 라이브러리 선정 기준(성능, 접근성, 유지보수성)이 정의되었는가?  
> - 이 단계에서 실제 화면 구현까지 포함하지 않는다는 범위 합의가 있는가?

## 목적

- 프로젝트의 공통 개발 환경 기준을 고정한다.
- UI 라이브러리 도입 여부와 선택 기준을 명시한다.
- 이후 디자인/라우팅 단계에서 재결정이 필요 없도록 초기 기술 의사결정을 문서화한다.

## 범위(포함/제외)

- 포함:
  - 런타임/패키지 매니저/코드 스타일 기준
  - UI 라이브러리 후보와 채택 기준 정의
  - 전역 스타일/토큰 연결 진입점 정의
- 제외:
  - 페이지별 UI 구현
  - 광고/SEO 기능 구현

## 선행조건

- 저장소가 실행 가능한 상태여야 한다.
- 팀 내 기본 개발 환경(OS, Node LTS, 패키지 매니저) 합의가 필요하다.

## 핵심 작업

- 개발 환경 표준(버전/명령어/로컬 실행 규칙) 정리
- UI 라이브러리 후보 비교 항목 정의
- 채택/보류 결정을 위한 의사결정 로그 포맷 준비
- 글로벌 스타일 확장 포인트(예: `app/globals.css`) 문서화

## 산출물

- 개발 환경 표준 체크리스트
- UI 라이브러리 선정 기록(채택/보류 사유 포함)
- 02단계로 전달할 디자인 토큰 연계 지점 목록

## 의사결정 포인트

- 패키지 매니저를 단일화할지 여부
- UI 라이브러리(또는 무라이브러리) 전략
- 컴포넌트 스타일링 방식(Tailwind 중심 vs 혼합)

## 리스크

- 라이브러리 의존성 증가로 번들/복잡도 확대 가능
- 초기 기준 부재 시 단계별 기술 편차 발생 가능

## 완료 기준

- 개발 환경 실행/빌드/린트 명령 기준이 문서에 명시됨
- UI 라이브러리 전략이 "채택/보류/대체안" 형태로 기록됨
- 다음 단계에서 재논의 없이 사용할 입력값이 정의됨

## 다음 단계 인계

- 정보구조/디자인 시스템에 반영할 기술 제약 조건
- 컬러/타이포 토큰 적용 가능한 전역 진입점 목록
