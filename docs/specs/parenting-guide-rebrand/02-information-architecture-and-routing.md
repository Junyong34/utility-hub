# Information Architecture And Routing

## 목표

리브랜딩된 사이트의 메뉴 구조, 허브 페이지, 라우트 정책을 고정한다.

## 상단 메뉴

1. 홈
2. 아이와 가볼 곳
3. 도구
4. 혜택·지원금
5. 블로그
6. 소개

이 순서는 `장소 → 도구 → 혜택` 우선순위를 반영한다.

## 사용자 노출 허브

- 홈
- 아이와 가볼 곳
- 도구
- 혜택·지원금
- 블로그
- 소개

## 라우트 정책

초기 라우트 제안:

- `/`
- `/places`
- `/tools`
- `/benefits`
- `/blog`
- `/about`

상세 글은 1차에서 현행 블로그 라우트 구조를 최대한 재사용한다.

- `/blog/[category]/[slug]`

이 방식의 이유:

1. 현재 블로그 시스템을 버리지 않는다.
2. 메뉴 구조는 새롭게 바꿀 수 있다.
3. 리브랜딩 비용을 낮춘다.

## 홈 정보구조

1. 히어로
2. 이번 주 인기 장소
3. 지역별 바로가기
4. 상황별 바로가기
5. 도구 모음
6. 혜택·지원금 모음
7. 최신 가이드

## 히어로 구성

필수 요소:

- H1
- 서브카피
- 검색 입력
- 빠른 필터

기본 빠른 필터:

- 서울
- 경기 북부
- 경기 남부
- 인천
- 12~24개월
- 24~48개월
- 5~7세
- 비 오는 날
- 실내
- 무료

## 탐색 축 우선순위

첫 탐색 축은 `지역 우선`이다.

2차 탐색 축:

- 연령
- 상황/테마
- 예산

## 허브 간 연결 규칙

- 홈은 세 허브를 타입별로 분리 노출
- 장소 허브는 도구와 혜택 링크를 항상 포함
- 도구 허브는 관련 장소 글 묶음을 반드시 포함
- 혜택 허브는 관련 도구와 장소 글을 같이 노출

## 현재 저장소 연결 지점

- 메뉴 설정: `components/layout/nav-config.ts`
- 블로그 카테고리/경로 처리: `lib/blog/posts.ts`
- 도구 허브 구성: `lib/tools/tool-config.ts`

## 결정해야 할 후속 항목

- 최종 브랜드명
- `/places`, `/benefits` 하위 라우트 방식
- 홈 검색의 실제 동작 범위

간단한 결론:

라우트는 새로 열되, 상세 글 구조는 최대한 재사용하는 것이 초기 구현에서 가장 안전하다.
