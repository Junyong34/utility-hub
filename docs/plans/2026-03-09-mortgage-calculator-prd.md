# PRD: 주택담보대출 비율 계산기 (DSR·DTI·LTV)

- 문서 상태: Draft v1
- 작성일: 2026-03-09
- 대상 서비스: Zento Tools
- 대상 경로: `/tools/mortgage-calculator`
- 문서 목적: 구현 중단/재개 상황에서도 동일한 기준으로 화면과 기능을 다시 만들 수 있도록 요구사항을 고정한다.

## 1. 배경

주택담보대출 관련 사용자는 DSR, DTI, LTV를 각각 따로 이해해야 하고, 실제 대출 가능성을 가늠하려면 세 지표를 함께 비교해야 한다. 그러나 기존 정보는 용어 설명과 계산식이 분리되어 있거나, 금융사 기준이 복잡해 일반 사용자가 입력해야 할 값과 결과 해석 포인트를 이해하기 어렵다.

이 도구는 단순 계산기보다 한 단계 더 나아가 다음을 동시에 제공하는 것을 목표로 한다.

1. DSR, DTI, LTV의 정의와 차이를 쉽게 설명한다.
2. 하나의 입력 세트로 3개 지표를 함께 계산한다.
3. 어떤 지표가 실제 한도를 더 제한하는지 병목을 보여준다.
4. 현재 계산 결과를 공유하고, 여러 케이스를 브라우저에 저장해 비교할 수 있게 한다.

## 2. 목표

1. 사용자가 1분 이내에 첫 계산을 완료할 수 있어야 한다.
2. 모바일에서 입력-계산-결과확인 흐름이 한 손 조작 중심으로 자연스러워야 한다.
3. 계산 결과뿐 아니라 왜 그런 결과가 나왔는지 설명을 함께 제공해야 한다.
4. 공유 링크로 현재 입력 상태를 다른 사람에게 그대로 전달할 수 있어야 한다.
5. 동일 사용자가 여러 케이스를 저장하고 나중에 다시 비교할 수 있어야 한다.

## 3. 비목표

1. 실제 금융기관의 승인 가능 여부를 확정하지 않는다.
2. 모든 금융상품과 예외 규정을 완전하게 재현하지 않는다.
3. 저장된 여러 케이스 전체를 공유하지 않는다.
4. 저장된 케이스의 수정 기능은 제공하지 않는다.
5. 사용자 인증, 서버 저장, 계정 동기화는 v1 범위에 포함하지 않는다.

## 4. 제품 포지셔닝

이 도구는 `법적/심사 확정 도구`가 아니라 `참고용 통합 계산기 + 설명 도구`다.

표시 원칙은 다음과 같다.

1. 계산값은 공개 기준과 사용자가 입력한 값에 근거한 참고치로 안내한다.
2. 금융기관별 내부 심사 기준, 인정소득, 지역 규제, 상품 조건에 따라 결과는 달라질 수 있음을 명시한다.
3. 결과 영역과 FAQ, 면책 고지에 같은 메시지를 반복 일관되게 노출한다.

## 5. 정보 구조와 라우팅

기본 구조는 `신규 Tool 1개 + 단일 페이지 탭형`이다.

- 메인 경로: `/tools/mortgage-calculator`
- 탭: `overview`, `dsr`, `dti`, `ltv`

탭 역할은 다음과 같다.

- `overview`: 세 지표 요약 비교, 병목 지표, 최종 해석
- `dsr`: DSR 정의, 공식, 스트레스 DSR 설명, 관련 입력
- `dti`: DTI 정의, 공식, DSR과 차이, 관련 입력
- `ltv`: LTV 정의, 공식, 선순위채권/임차보증금 반영 설명

라우팅 전략은 개별 페이지 분리보다 단일 페이지 탭형을 우선한다. 이유는 다음과 같다.

1. 사용자가 세 지표를 함께 비교해야 하는 문제 구조와 맞다.
2. 공유 URL을 `nuqs` 상태값으로 단일하게 관리하기 쉽다.
3. 모바일에서 탭 전환과 입력 유지 경험이 더 자연스럽다.
4. 기존 Tool 구조와 SEO 설정을 무리 없이 확장할 수 있다.

## 6. 핵심 사용자

### 1차 사용자

- 생애최초 또는 일반 실수요자
- 주택 매수 전 대출 가능 범위를 대략 확인하려는 사용자
- DSR, DTI, LTV 차이를 잘 모르는 사용자

### 2차 사용자

- 부동산 중개/상담 과정에서 빠르게 케이스를 비교하려는 사용자
- 배우자/가족과 결과를 공유해야 하는 사용자
- 여러 시나리오를 저장해 비교하려는 사용자

## 7. 사용자 시나리오

### 시나리오 A: 빠른 1회 계산

1. 사용자는 페이지에 진입한다.
2. 기본 입력 필드를 채운다.
3. 계산 버튼을 누른다.
4. 인라인 요약 카드에서 DSR/DTI/LTV/병목 결과를 본다.
5. 상세 결과를 BottomSheet에서 확인한다.
6. 현재 계산 결과 링크를 복사해 공유한다.

### 시나리오 B: 여러 케이스 비교

1. 사용자는 금리/소득/대출금액을 바꿔가며 여러 번 계산한다.
2. 각 결과를 저장한다.
3. 저장된 케이스 목록에서 제목/날짜/핵심 결과를 비교한다.
4. 필요 없는 케이스는 삭제한다.

### 시나리오 C: 용어 이해

1. 사용자는 계산 결과만으로 이해가 어려워한다.
2. 탭별 설명 영역과 FAQ를 읽는다.
3. 왜 DSR과 DTI가 다른지, 왜 LTV에 선순위채권이 필요한지 이해한다.

## 8. 기능 요구사항

### 8.1 입력 구조

입력은 `기본 입력`과 `고급 입력`으로 나눈다.

#### 기본 입력

- 연소득
- 주택가액
- 희망 대출금액
- 대출금리
- 대출만기(년)
- 상환방식
- 기존대출 월 원리금 합계
- 기타부채 월 이자 합계

#### 고급 입력

- 선순위채권 금액
- 임차보증금 또는 최우선변제금 반영 금액
- 지역유형
- 상품유형
- 생애최초/실수요자 여부
- 스트레스 DSR 반영 여부
- 스트레스 금리 가산값
- 사용자가 직접 조정 가능한 DSR/DTI/LTV 한도값

기본 원칙은 다음과 같다.

1. 첫 진입 시 기본 입력만 보여준다.
2. 고급 입력은 아코디언 또는 접기 영역으로 분리한다.
3. 대부분 사용자는 기본 입력만으로 계산을 완료할 수 있어야 한다.
4. 고급 입력은 “왜 필요한지” 짧은 설명을 함께 보여준다.

### 8.2 계산 방식

#### DSR

- 정의: 모든 가계대출의 연간 원리금 상환액이 연소득에서 차지하는 비율
- 기본 공식: `DSR = (본건 주담대 연 원리금 + 기존 모든 가계대출 연 원리금) / 연소득 × 100`
- 참고 표시: 은행권 40%, 비은행권 50% 등은 기본 프리셋으로 제공하되 사용자가 조정 가능하게 한다.
- 스트레스 DSR: `적용금리 = 입력 금리 + 스트레스 금리`를 적용한 참고 계산 결과를 별도 배지로 표시한다.

#### DTI

- 정의: 주택담보대출 원리금과 기타부채 이자상환 추정액이 연소득에서 차지하는 비율
- 기본 공식: `DTI = (본건 주담대 연 원리금 + 기타부채 연 이자) / 연소득 × 100`
- 설명 포인트: DSR은 모든 가계대출 원리금을 보지만, DTI는 본건 주담대와 기타부채 이자를 중심으로 본다는 차이를 강조한다.

#### LTV

- 정의: 주택 담보가치 대비 대출금액 비율
- 기본 공식: `기본 LTV = 희망 대출금액 / 주택가액 × 100`
- 조정 공식: `조정 LTV = (희망 대출금액 + 선순위채권 + 임차보증금 등) / 주택가액 × 100`
- 설명 포인트: 실제 대출 가능액은 선순위 권리관계와 지역/상품 조건에 따라 달라질 수 있음을 명시한다.

#### 병목 한도

- 제품은 단순 비율 표시를 넘어 “어떤 지표가 가장 보수적으로 작동하는지”를 보여줘야 한다.
- 병목 한도는 `DSR 역산 가능 대출액`, `DTI 역산 가능 대출액`, `LTV 역산 가능 대출액` 중 가장 작은 값을 기준으로 잡는다.
- 결과 카드에 `현재 케이스의 제한 요인: DSR / DTI / LTV`를 명확히 표시한다.

### 8.3 결과 표시

#### 인라인 요약 카드

- DSR 결과
- DTI 결과
- LTV 결과
- 최종 병목 지표
- 참고 한도 대비 여유 또는 초과 상태
- 핵심 해석 한 줄

#### BottomSheet 상세 결과

- 계산 요약
- 사용된 입력값
- 계산식 요약
- 한도 역산 결과
- 복사 버튼
- 저장 버튼
- 공유 링크 복사 버튼
- 면책 고지

결과 시트는 모바일 중심으로 설계하되 데스크톱에서도 동일한 내용 구조를 유지한다.

### 8.4 저장 기능

저장은 `localStorage`에만 보관한다.

저장 규칙:

- 최대 20개 케이스
- 최신순 정렬
- 케이스 수정 불가
- 삭제만 가능
- 저장 시 결과 스냅샷을 함께 보관
- 저장 시점의 입력값, 결과값, 생성시간, 병목 지표를 함께 저장

목록에서 보여줄 항목:

- 저장 시간
- 케이스 요약 제목
- 핵심 입력값 일부
- DSR/DTI/LTV 요약
- 병목 지표
- 삭제 버튼

### 8.5 공유 기능

공유는 `nuqs` 상태값을 사용한다.

공유 원칙:

- 현재 입력값만 URL에 반영
- 현재 선택 탭도 URL에 반영
- 저장된 여러 케이스는 공유하지 않음
- 공유 링크를 열면 동일 입력 상태로 계산 화면이 복원되어야 함

### 8.6 삭제 기능

- 저장 케이스는 개별 삭제 가능
- 전체 삭제는 v1에서 선택 사항
- 삭제 전 간단한 확인 UX 필요
- 삭제 후 토스트 또는 인라인 피드백 필요

## 9. UX/UI 요구사항

### 9.1 모바일 우선 레이아웃

화면 순서는 다음을 기본으로 한다.

1. 페이지 헤더
2. 계산기 소개와 핵심 설명
3. 탭 전환
4. 기본 입력 폼
5. 고급 입력 접기 영역
6. 계산 CTA
7. 인라인 요약 결과
8. 설명 콘텐츠
9. FAQ
10. 저장 케이스 목록

### 9.2 입력 UX

- 숫자 입력은 모바일 키패드 친화적으로 구성
- 금액 입력은 단위 힌트 제공
- `만원`, `억원`, `원` 등 보조 단위 표시 고려
- placeholder에 현실적인 예시 제공
- 필드별 에러 문구는 짧고 명확하게 제공
- 필수값 누락 시 어디를 채워야 하는지 즉시 알 수 있어야 함

### 9.3 탭 UX

- 탭은 상단 고정 또는 시인성 높은 세그먼트 UI 사용
- 각 탭은 같은 입력 세트를 공유하되, 강조 설명과 결과 해석만 달라질 수 있음
- `overview` 탭은 요약 중심
- 개별 탭은 해당 지표를 더 자세히 설명

### 9.4 결과 UX

- 계산 버튼을 누른 직후 결과 영역으로 자연스럽게 이동 또는 포커스 이동
- 결과 요약은 숫자만이 아니라 해석 문장을 함께 제공
- BottomSheet는 결과 재확인 진입점으로 재오픈 가능해야 함
- 복사 후에는 사용자가 성공 여부를 명확히 인지해야 함

## 10. 콘텐츠 요구사항

### 10.1 설명 영역

설명 영역은 계산기 못지않게 중요하다. 다음 내용을 포함한다.

- DSR, DTI, LTV의 한 줄 정의
- 세 용어의 차이 비교
- 각각 어떤 상황에서 중요해지는지
- 왜 실제 금융기관 결과와 차이가 날 수 있는지
- 스트레스 DSR이 무엇인지
- 선순위채권/임차보증금이 LTV에 왜 영향을 주는지

### 10.2 FAQ 초안

FAQ는 최소 8개 이상으로 구성한다.

예상 문항:

- DSR, DTI, LTV는 무엇이 다른가요?
- 은행 상담 결과와 왜 다를 수 있나요?
- 스트레스 DSR은 무엇인가요?
- DTI에는 왜 기타 대출 이자만 들어가나요?
- LTV 계산에 선순위채권은 왜 포함되나요?
- 소득은 세전 기준인가요, 세후 기준인가요?
- 저장한 케이스는 다른 사람에게 공유되나요?
- 이 계산기로 실제 대출 승인 여부를 알 수 있나요?

### 10.3 면책 문구

- 참고용 계산기임을 명시
- 실제 한도/금리/승인 여부는 금융기관 심사에 따름
- 정책과 규제는 변경될 수 있음
- 최신 공공기관 자료 확인을 권장

## 11. SEO/GEO 요구사항

- Tool 메타는 기존 `tool-config` 구조를 따른다.
- 구조화 데이터는 `WebPage`, `Breadcrumb`, `SoftwareApplication`, `FAQPage`, `HowTo`를 포함한다.
- 주요 키워드:
  - 주택담보대출 계산기
  - DSR 계산기
  - DTI 계산기
  - LTV 계산기
  - 주담대 비율 계산
  - 스트레스 DSR 계산기
- FAQ와 설명 영역은 검색 유입과 AI 검색 인용을 고려해 답변 우선 구조로 작성한다.
- 계산 공식, 해석 기준, 주의사항, 출처를 명확히 노출한다.

## 13. 성능 및 접근성 요구사항

- 모바일에서 핵심 인터랙션은 지연 없이 반응해야 함
- 결과 계산은 클라이언트에서 즉시 수행 가능해야 함
- BottomSheet는 키보드 접근성과 닫기 동작을 보장해야 함

- 색상만으로 상태를 구분하지 않음
- 에러/성공 피드백은 텍스트를 함께 제공

## 18. 최종 결론

이 PRD의 기본안은 `단일 페이지 탭형 + 통합 시나리오 계산 + 모바일 우선 UX + BottomSheet 결과 + nuqs 공유 + localStorage 케이스 저장`이다.

제품은 단순 계산기보다 `설명 가능한 계산기`에 가깝게 설계해야 하며, 구현 시 가장 중요한 기준은 다음 세 가지다.

1. 사용자가 쉽게 입력할 수 있어야 한다.
2. 결과를 쉽게 이해할 수 있어야 한다.
3. 현재 계산을 쉽게 공유하고 여러 케이스를 쉽게 비교할 수 있어야 한다.
