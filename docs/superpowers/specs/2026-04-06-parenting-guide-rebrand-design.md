# Parenting Guide Rebrand Design Spec

**Goal:** `zento.kr`를 유지한 채, 부모가 `아이와 갈 곳`, `실전 도구`, `혜택·지원금`을 빠르게 찾는 실용형 육아 사이트로 재정의한다.

**Status:** Approved design direction, ready for implementation planning.

**Source Inputs**

- 사용자 결정:
  - 핵심 독자: 수도권 1~7세 자녀 부모
  - 핵심 과업: 아이와 갈 곳 찾기
  - 홈 구조: 검색 + 큐레이션 혼합형
  - 첫 탐색 축: 지역 우선
  - 론칭 범위: 수도권 우선, 이후 광역시와 전국으로 확장
  - 콘텐츠 단위: SEO 글 + 표준 카드 혼합형
  - 보조 축 우선순위: 도구 우선, 혜택·지원금은 별도 타입으로 분리 노출
  - 브랜드 톤: 실용형
  - 사업 목적: 검색 유입 + 도구 사용 연결
- 현재 프로젝트 구조:
  - 내비게이션 소스: `components/layout/nav-config.ts`
  - 도구 소스 오브 트루스: `lib/tools/tool-config.ts`
  - 블로그 로더와 카테고리 매핑: `lib/blog/posts.ts`

---

## 1. Product Thesis

이 사이트는 육아 커뮤니티가 아니다.

이 사이트는 `주말이나 쉬는 날 아이와 어디 갈지 빠르게 정리하고`, 그 과정에서 `비용과 혜택까지 같이 판단하게 만드는 실전 가이드`다.

핵심 경험은 아래 순서로 설계한다.

1. 사용자는 지역과 조건으로 갈 곳을 찾는다.
2. 관련 글 안에서 표준 장소 카드와 비교 정보를 본다.
3. 필요하면 관련 계산 도구로 이동한다.
4. 같은 맥락의 혜택·지원금 글로 이어진다.
5. 다시 다른 지역/상황 글로 탐색을 확장한다.

이 흐름이 사이트의 기본 퍼널이다.

## 2. Brand Strategy

### 2.1 Brand Positioning

대외 포지션은 `아이와 갈 곳, 조건별로 빠르게 찾는 실용형 육아 가이드`로 고정한다.

커뮤니티성, 감성 에세이, 후기 중심 브랜드로 가지 않는다.

### 2.2 Naming and Domain

도메인은 계속 `zento.kr`를 사용한다.

사용자에게 노출되는 서비스 브랜드는 별도 육아 브랜드로 운영한다. 네이밍 확정 전까지 문서와 화면에서는 `육아 실전 가이드 by Zento`를 임시 표기로 사용한다.

핵심 규칙은 아래와 같다.

1. 도메인은 인프라다.
2. 사용자 노출 브랜드는 새 육아 브랜드로 일관되게 보여준다.
3. `Zento`는 후행 표기 또는 보조 표기로만 남긴다.
4. 메타데이터, OG, 헤더, 푸터, 카피는 혼용하지 않는다.

### 2.3 Tone of Voice

톤은 `실용형`으로 고정한다.

카피 원칙:

- 바로 찾게 한다.
- 바로 비교하게 한다.
- 바로 결정하게 한다.
- 과장하지 않는다.
- 육아 불안을 자극하지 않는다.

예시:

- 좋은 톤: `비 오는 날 수원에서 아이와 갈 만한 곳 8곳`
- 나쁜 톤: `엄마들이 꼭 알아야 할 최고의 비밀 장소`

## 3. Benchmark Translation

벤치마크는 `맘맘`의 서비스 모델이 아니라, `탐색 방식`이다.

가져올 것:

1. 지역과 연령 맥락에서 시작하는 탐색 구조
2. 사용 목적이 바로 드러나는 홈 진입
3. 부모가 당장 필요한 카테고리를 전면 배치하는 정보 구조

가져오지 않을 것:

1. 로그인 중심 구조
2. 거래, 입점, 핫딜, 멤버십 운영 모델
3. 커머스 전환을 위한 과도한 카드 밀도

우리의 차별점:

1. 거래형이 아니라 검색 유입형이다.
2. 장소만 보여주는 것이 아니라, 비용 판단과 도구 사용까지 연결한다.
3. 글 구조 안에 비교표, 체크리스트, 표준 카드, 계산기 링크를 넣는다.
4. 부모가 `읽고 끝나는 사이트`가 아니라 `읽고 계산하고 다시 찾는 사이트`가 되게 한다.

## 4. Target Audience and Jobs To Be Done

### 4.1 Primary Audience

첫 론칭 타깃은 `수도권에 거주하는 1~7세 자녀 부모`다.

특히 아래 상황에 있는 부모를 상정한다.

- 주말마다 아이와 갈 곳을 급하게 정해야 하는 부모
- 너무 멀지 않은 곳을 우선 찾는 부모
- 실내/무료/저예산 조건을 자주 거는 부모
- 지원금이나 비용 판단도 함께 보고 싶은 부모

### 4.2 Jobs To Be Done

핵심 JTBD:

- `이번 주말에 아이와 어디 가지?`

보조 JTBD:

- `이 나이에 맞는 곳인가?`
- `비 오는 날에도 가능한가?`
- `너무 비싸지 않은가?`
- `관련 혜택이나 절약 방법이 있나?`
- `이 비용을 대충이라도 계산할 수 있나?`

## 5. Launch Scope and Expansion Path

### 5.1 Phase A, Launch Scope

1차 론칭 범위는 수도권만 다룬다.

- 서울
- 경기 북부
- 경기 남부
- 인천

### 5.2 Phase B, Expansion

2차 확장에서는 광역시와 주요 도시를 붙인다.

- 부산
- 대구
- 대전
- 광주
- 세종
- 주요 광역 생활권

### 5.3 Phase C, National Scale

3차 확장에서는 전국 구조를 연다.

전국 구조를 먼저 열어두고 나중에 채우는 방식은 금지한다. 초기에는 깊이를 우선한다.

## 6. Information Architecture

### 6.1 Top Navigation

상단 메뉴는 아래 순서로 고정한다.

1. 홈
2. 아이와 가볼 곳
3. 도구
4. 혜택·지원금
5. 블로그
6. 소개

이 순서는 `장소 → 도구 → 혜택` 우선순위를 반영한다.

### 6.2 Top-Level Hubs

사용자 노출 허브는 아래 세 개다.

1. `아이와 가볼 곳`
2. `도구`
3. `혜택·지원금`

`블로그`는 운영 단위이자 아카이브 허브로 유지한다.

### 6.3 Homepage Structure

홈은 `검색 + 큐레이션 혼합형`으로 고정한다.

섹션 순서:

1. 히어로
2. 이번 주 인기 장소
3. 지역별 바로가기
4. 상황별 바로가기
5. 도구 모음
6. 혜택·지원금 모음
7. 최신 가이드

### 6.4 Hero Definition

히어로는 아래 정보를 포함해야 한다.

- H1: `아이와 갈 곳, 조건별로 빠르게 찾으세요`
- 서브카피: `지역, 연령, 날씨, 예산 기준으로 주말 계획을 바로 정리`
- 검색 입력
- 빠른 필터

빠른 필터 기본 세트:

- 서울
- 경기 북부
- 경기 남부
- 인천
- 12~24개월
- 24~48개월
- 5~7세
- 비 오는 날
- 실내
- 무료

### 6.5 Route Policy

초기 구현에서는 `사용자 노출 허브`와 `콘텐츠 저장 구조`를 분리한다.

정책은 아래와 같다.

1. 홈은 `/`
2. `아이와 가볼 곳` 허브는 `/places`
3. `혜택·지원금` 허브는 `/benefits`
4. `도구` 허브는 기존 `/tools`를 유지
5. 블로그 상세 글은 1차에서는 현행 `/blog/[category]/[slug]` 구조를 유지할 수 있다

이 방식의 이유:

1. 현재 저장소의 블로그 시스템을 최대한 재사용할 수 있다.
2. 메뉴와 사용자 경험은 육아 사이트 기준으로 재정의할 수 있다.
3. 상세 글 경로 전체를 한 번에 갈아엎지 않아도 된다.

## 7. Content System

### 7.1 Content Type Model

콘텐츠는 `SEO 글 중심 + 표준 카드 블록` 방식으로 운영한다.

기본 규칙:

1. 검색 유입을 받는 단위는 글이다.
2. 글 안에는 반복 가능한 표준 카드 블록을 넣는다.
3. 카드 데이터는 나중에 장소 DB로 확장 가능한 구조를 가진다.

### 7.2 Primary Content Buckets

블로그는 아래 세 축으로 운영한다.

1. 지역별 가이드
2. 테마별 가이드
3. 문제 해결형 가이드

예시:

- `수원 24~48개월 실내 체험 7곳`
- `서울 무료 키즈 시설 모음`
- `비 오는 날 성남에서 갈 만한 곳`

### 7.3 Secondary Taxonomy

글과 카드에는 아래 메타 축을 붙인다.

- 지역
- 연령대
- 실내/야외
- 예산대
- 날씨 적합도
- 테마
- 동반 조건

`동반 조건` 예시:

- 조부모 동반 가능
- 유모차 이동 편함
- 주차 편함

### 7.4 Standard Place Card Schema

표준 장소 카드에는 최소 아래 필드를 가진다.

- 장소명
- 지역
- 권장 연령
- 실내/야외
- 가격대
- 핵심 포인트 한 줄

가능하면 같이 넣을 필드:

- 주차
- 수유실
- 유모차
- 우천 대체 가능 여부
- 체류 권장 시간

### 7.5 Editorial Rules

글 하나를 발행할 때 반드시 같이 해야 하는 작업:

1. 관련 지역 글 링크 1개 이상
2. 관련 도구 링크 1개 이상
3. 관련 혜택·지원금 글 링크 1개 이상
4. 표준 카드 블록 사용
5. 메타 필드 입력

### 7.6 Non-Goals for Content

초기에는 아래를 하지 않는다.

1. 사용자 후기 수집 기능
2. 댓글 운영
3. 장소 제보 플랫폼화
4. 전국 랜딩 페이지 동시 오픈
5. 감성 육아 일기형 콘텐츠 대량 발행

## 8. Tool System

### 8.1 Tool Positioning

도구는 메인 축이다.

하지만 홈의 첫 메시지를 잡아먹으면 안 된다. 사용자는 먼저 갈 곳을 찾고, 이후 판단을 위해 도구로 넘어가야 한다.

### 8.2 Launch Priority

1차 우선순위는 아래와 같다.

1. 연령/개월수 계산기
2. 성장 계산기
3. 자녀 비과세 증여 계산기
4. 가족 나들이 예산 계산기
5. 키즈카페·체험 월예산 계산기
6. 어린이집·유치원 비용 계산기

### 8.3 Tool Rules

도구 운영 규칙:

1. 각 도구는 관련 글 2~3개 이상과 연결한다.
2. 도구 페이지는 독립 SEO 랜딩으로도 읽혀야 한다.
3. 의료 진단처럼 오해될 수 있는 계산기는 초기 범위에서 제외한다.
4. 결과 화면에는 행동 유도 링크를 둔다.

행동 유도 링크 예시:

- 관련 장소 글 보기
- 관련 혜택 글 보기
- 비슷한 계산기 보기

## 9. Benefits and Support System

### 9.1 Coverage Areas

혜택·지원금 허브는 아래 3축으로 운영한다.

1. 정부지원
2. 지역 혜택
3. 절약 가이드

### 9.2 Writing Pattern

혜택 글은 공지 나열형으로 쓰지 않는다.

반드시 아래 질문에 답하는 구조로 쓴다.

1. 누가 받을 수 있나
2. 언제 신청하나
3. 얼마나 도움이 되나
4. 실제로 어디에 쓰이나
5. 같이 보면 좋은 계산기나 가이드가 있나

## 10. Discovery and SEO Strategy

### 10.1 Query Pattern

초기 SEO는 아래 유형의 검색어를 우선 잡는다.

1. `지역 + 연령 + 실내/야외 + 장소`
2. `지역 + 무료 + 아이와 갈 곳`
3. `날씨/상황 + 지역 + 아이와 갈 곳`
4. `육아 비용/증여/예산 + 계산기`

### 10.2 Internal Linking Strategy

내부 링크는 아래 방향으로 깐다.

- 장소 글 → 관련 도구
- 장소 글 → 관련 혜택
- 도구 → 관련 장소 글
- 혜택 글 → 관련 도구와 장소 글

핵심은 단일 글 유입을 `다음 페이지 소비`로 연결하는 것이다.

### 10.3 Current Repo Fit

현재 저장소는 블로그와 도구 허브가 이미 분리되어 있어 구조적으로 맞는다.

후속 구현 시 확인해야 할 기준 파일:

- `components/layout/nav-config.ts`
- `lib/tools/tool-config.ts`
- `lib/blog/posts.ts`

블로그 카테고리와 메타 필드 체계는 현행 구조를 확장해서 처리한다.

## 11. Success Metrics

초기 성공 기준은 아래 세 가지다.

1. 검색 유입이 늘어나는가
2. 글 유입 후 도구 클릭이 발생하는가
3. 사용자 한 명이 2페이지 이상 이동하는가

정량 예시는 구현 계획 단계에서 확정한다.

이 스펙 단계에서는 측정 축만 고정한다.

## 12. Risks and Guardrails

### 12.1 Main Risks

1. 전국 욕심으로 초기 범위가 무너질 수 있다.
2. 글 제목이 지역/연령/상황 축 없이 평범해질 수 있다.
3. 도구가 장소 경험과 끊어지면 별도 섬처럼 보일 수 있다.
4. 혜택 글이 공지 게시판처럼 보이면 재방문성이 떨어진다.

### 12.2 Guardrails

1. 론칭 전까지 전국 구조를 열지 않는다.
2. 모든 글은 최소 하나의 지역과 하나의 조건 축을 가진다.
3. 모든 도구는 관련 글과 연결된다.
4. 홈에서는 장소, 도구, 혜택을 서로 다른 타입으로 분리 노출한다.

## 13. Non-Goals

이번 설계 범위 밖 항목:

1. 로그인
2. 댓글
3. 회원 등급
4. 거래/예약/커머스
5. 후기 플랫폼
6. 앱 전용 기능
