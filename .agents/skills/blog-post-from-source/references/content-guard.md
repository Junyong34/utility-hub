# 콘텐츠 개인정보 보호 가이드 (content-guard)

## 목적
링크/원문 텍스트 기반으로 글을 작성할 때 개인정보(PII)와 민감 비즈니스 정보가 본문에 노출되지 않도록 가이드를 정리한다.

## 우선 마스킹 패턴
아래 항목은 기본 마스킹 대상이다.

- 전화번호: `010-1234-5678`, `02-123-4567`, `02 123 4567` 형태
- 이메일: `name@example.com`
- 계좌/카드/사업자번호 형태
- 주민등록번호 의심 패턴: `######-#######`
- 여권/여권번호/면허번호처럼 고유 ID로 보이는 연속 숫자/영문 조합
- 실주소(건물/동/호수까지 구체적으로 공개된 항목)
- URL의 개인 URL 파라미터가 토큰/유저ID/세션ID처럼 보이는 값

## 처리 규칙
1. 감지 즉시 제거 또는 익명화
- 사람 이름은 `○○씨` 같은 일반화, 기업명은 과도하게 구체적이면 `해당 업체`로 치환.
- 번호 정보는 `연락처는 공개되지 않음` 또는 `연락 가능한 채널은 생략`으로 통일.
- 특정인 신원 식별이 가능한 경우 초반에 `개인정보는 비공개 처리했습니다` 배지를 표시.

2. 사실 유지 원칙
- 정보 삭제로 인해 문맥이 깨지면 문장을 `이 항목은 공개 가능한 범위 밖이므로 일반화했습니다`로 대체.
- 수치/날짜가 인용이라면 원문에서 확인 가능한 범위만 남기고 나머지는 생략.

3. 예외
- 회사명, 공시명, 공개된 제품명은 일반적으로 유지.
- 수치(가격, 기간, 기간별 실적)는 공개 출처가 있으면 유지.

## 신뢰도 표기
- 출처가 불확실한 내용은 본문 근처에 `This may not be accurate.` 를 붙인다.
- 사실 기반 정보는 각 단락 말미에 출처 문구를 1개 이상 배치한다.

## 출력 금지
- 실명/연락처/계좌/이메일/식별자와 같은 항목의 부분 노출.
- "공개용으로 이미 검증됨" 이라는 확인 없는 단정.
- 사용자의 원문을 인용한 뒤 원문 그대로 재노출.

## 빠른 점검 체크리스트
- [ ] 이름, 연락처, 계정, 주소 중 식별 정보가 그대로 남지 않았는가
- [ ] 숫자 문자열이 너무 구체적으로 개인을 특정하지 않도록 일반화되었는가
- [ ] `개인정보는 비공개 처리했습니다` 라벨이 필요한 곳에 들어갔는가
- [ ] 추정 문장에는 "이 정보는 확정하기 어렵습니다" 또는 `This may not be accurate.` 가 있는가
