# blog-seo-writer: 한국어 블로그 SEO 작성/검수 스킬

## 소개

이 스킬은 한국어 블로그 포스트를 작성하거나 검수할 때, 검색 엔진 노출 가능성을 높이기 위한 실무 지침을 제공합니다.

- 목적: 구조화된 글 구성, FAQ, JSON-LD, 최신성/내부 링크 전략, Humanizer 기반 최종 문장 검수
- 적용 범위: SEO 작성/검수
- 비적용 범위: 키워드 리서치 자동화, 발행 스케줄 자동화, 성과 리포트 자동화

## 입력 체크리스트

### 필수 입력

1. 포스트 초안 텍스트
2. 주제
3. 핵심 키워드
4. 변형 키워드
5. 작성자 전문성 정보(경력, 자격, 실무 이력)
6. 내부 링크 후보(링크 URL + 앵커 텍스트)
7. 최신 업데이트 날짜(YYYY-MM-DD)

### 선택 입력

- 타깃 독자
- 원하는 문체/톤
- 반드시 포함해야 할 제품/서비스 정보
- 금지 표현 또는 법적/브랜드 제약

### 입력 누락 시 규칙

- 필수 입력이 없으면 작성을 강행하지 말고 먼저 누락 항목을 요청한다.
- 사실 근거가 부족하면 추측으로 채우지 않는다.

## 출력 계약

최종 결과는 항상 아래 5개 섹션을 포함한다.

1. 개선 본문
2. FAQ(최소 3개 문항, 짧은 글은 1-2개 가능)
3. JSON-LD(`BlogPosting` + `FAQPage`)
4. Humanizer 검수 결과
5. 최종 체크리스트

**참고:** 블로그 주제와 내용에 따라 선택적 섹션(TL;DR, 속성 테이블 등)은 포함/제외될 수 있다.

## SEO 작성 워크플로우

**중요:** 아래 단계는 블로그 주제와 내용에 따라 선택적으로 적용한다. 모든 단계를 반드시 적용할 필요는 없으며, 주제의 특성에 맞게 필요한 단계만 선택하여 적용한다.

### 단계별 적용 가이드

| 단계 | 필수 여부 | 적용 제외 케이스 |
|------|----------|-----------------|
| 1. 제목 최적화 | 필수 | 없음 (모든 포스트에 적용) |
| 2. TL;DR 요약 | 권장 | 짧은 글(1000자 이하), 단순 공지사항 |
| 3. 도입 핵심 정의 | 필수 | 없음 (모든 포스트에 적용) |
| 4. 리스트/테이블 구조화 | 권장 | 서사형 에세이, 스토리텔링 중심 글 |
| 4-1. 장소 속성 테이블 | 조건부 | 장소/매장/식당 소개가 아닌 경우 |
| 4-2. 제품 리뷰 속성 테이블 | 조건부 | 제품/서비스 리뷰가 아닌 경우 |
| 5. AI 크롤링 최적화 | 권장 | 개인 에세이, 감상문, 창작물 |
| 6. FAQ 섹션 | 필수 | 없음 (최소 3개 작성) |
| 7. 작성자 전문성 표기 | 권장 | 개인 일기, 가벼운 일상 포스트 |
| 8. 주제 집중 전략 | 필수 | 없음 (모든 포스트에 적용) |

### 주제별 적용 예시

#### 케이스 1: 기술 튜토리얼 (예: "Next.js SEO 설정 방법")
**적용:** 1, 2, 3, 4, 6, 7, 8
**제외:** 4-1(장소 테이블), 4-2(제품 테이블), 5(구체적 속성 표현 불필요)

#### 케이스 2: 식당/장소 리뷰 (예: "강남 맛집 ○○ 레스토랑 후기")
**적용:** 1, 2, 3, 4, 4-1(장소 테이블), 4-2(평가 테이블), 5, 6, 8
**제외:** 7(작성자 전문성 선택)

#### 케이스 3: 제품 리뷰 (예: "맥북 프로 M3 사용 후기")
**적용:** 1, 2, 3, 4, 4-2(제품 속성 테이블), 5, 6, 8
**제외:** 4-1(장소 테이블), 7(작성자 전문성 선택)

#### 케이스 4: 개인 에세이/회고 (예: "2024년 개발자로 살아가며 배운 것들")
**적용:** 1, 3, 6, 8
**제외:** 2(TL;DR 선택), 4(구조화 선택), 4-1, 4-2, 5, 7

#### 케이스 5: 개념 설명 (예: "RESTful API란 무엇인가?")
**적용:** 1, 2, 3, 4, 6, 7, 8
**제외:** 4-1(장소 테이블), 4-2(제품 테이블), 5(구체적 속성 표현 불필요)

#### 케이스 6: 짧은 팁/공지 (500자 이하)
**적용:** 1, 3, 6(FAQ 1-2개), 8
**제외:** 2(TL;DR), 4, 4-1, 4-2, 5, 7

### 1단계: 제목 최적화(질문형)

- 제목을 질문형으로 바꾼다.
- 핵심 키워드를 제목 앞부분에 자연스럽게 포함한다.
- 과장/낚시형 문구 대신 검색 의도와 일치하는 질문을 만든다.

예시:

- 변경 전: `Next.js SEO 설정 방법`
- 변경 후: `Next.js에서 SEO를 제대로 설정하려면 무엇부터 해야 할까?`

### 2단계: TL;DR 요약 작성

- 포스트 상단에 TL;DR(Too Long; Didn't Read) 섹션을 추가한다.
- 핵심 내용을 3-5줄로 간결하게 요약한다.
- 독자가 전체 내용을 빠르게 파악할 수 있도록 작성한다.

권장 패턴:

1. 주제의 핵심 결론 1-2문장
2. 주요 포인트 2-3개 항목
3. 예상 독자 이익 1문장

예시:

```markdown
## TL;DR

Next.js SEO 설정의 핵심은 메타 태그, sitemap.xml, robots.txt 설정입니다. 이 글에서는 실무에서 바로 적용 가능한 설정 방법과 JSON-LD 구조화 데이터 적용법을 다룹니다. 올바른 SEO 설정으로 검색 엔진 노출을 최대 3배까지 높일 수 있습니다.
```

### 3단계: 도입 3줄 내 핵심 정의

- 글 시작 3줄 안에 주제의 핵심 정의를 명시한다.
- 독자가 "이 글이 무엇을 해결하는지" 즉시 이해할 수 있어야 한다.

권장 패턴:

1. 주제 정의 1문장
2. 왜 중요한지 1문장
3. 이 글에서 다룰 범위 1문장

### 4단계: 리스트/테이블 구조화 및 속성 정보 표 작성

- 긴 단락을 그대로 유지하지 말고, 리스트 또는 테이블로 재구성한다.
- 비교, 장단점, 절차, 체크포인트는 표 형태를 우선 검토한다.
- 본문에는 최소 1개 이상의 리스트 또는 테이블을 포함한다.

#### 장소/지역 소개 시 필수 속성 테이블

장소, 지역, 매장, 식당 등을 소개할 때는 AI가 구조화된 정보로 인식할 수 있도록 속성 정보를 표로 작성한다.

필수 포함 항목:

- 이름(상호명)
- 위치(주소)
- 전화번호
- 영업시간
- 주차 가능 여부
- 가격대
- 대표 메뉴/특징

예시:

```markdown
| 항목 | 정보 |
|------|------|
| 상호명 | ○○ 레스토랑 |
| 위치 | 서울시 강남구 ○○동 123-45 |
| 전화번호 | 02-1234-5678 |
| 영업시간 | 평일 11:00-22:00, 주말 10:00-23:00 |
| 주차 | 가능 (발렛파킹 제공) |
| 가격대 | 1인당 30,000-50,000원 |
| 특징 | 어두운 조명, 테이블 간격 넓음, 이유식 의자 제공 |
```

#### 제품/서비스 리뷰 시 속성 테이블

제품, 서비스, 경험을 리뷰할 때는 구조화된 속성 정보를 표로 작성한다.

예시 (식당 리뷰):

```markdown
| 평가 항목 | 평가 내용 |
|----------|----------|
| 분위기 | 어두운 조명, 차분한 인테리어 |
| 좌석 배치 | 테이블 간격 넓음, 프라이버시 확보 |
| 서비스 | 친절함, 이유식 의자 및 아기 물티슈 제공 |
| 야채 신선도 | 매우 신선, 샐러드바 운영 |
| 고기 품질 | 1++ 등급, 입에서 살살 녹음 |
| 가성비 | 높은 편, 세트 메뉴 추천 |
```

### 5단계: AI 크롤링 최적화를 위한 구조화 작성

AI 검색 엔진(ChatGPT, Perplexity 등)이 콘텐츠를 효과적으로 크롤링하고 참조 링크로 활용할 수 있도록 작성한다.

#### 핵심 원칙

1. **구조화된 정보 우선**: 산문 형태보다 표, 리스트, 명확한 속성 정보를 우선한다.
2. **구체적 속성 표현**: 모호한 표현 대신 구체적이고 측정 가능한 표현을 사용한다.
3. **검색 의도 매칭**: 사용자가 실제로 검색할 질문에 대한 명확한 답변을 제공한다.

#### 구체적 속성 표현 가이드

모호한 표현을 피하고 구체적으로 작성한다:

**나쁜 예시:**
- "분위기가 좋다" → 구체성 부족
- "맛있다" → 주관적, 측정 불가
- "괜찮은 곳" → 모호함

**좋은 예시:**
- "어두운 조명의 차분한 분위기, 테이블 간격 2m 이상 확보"
- "야채가 매일 새벽 직송으로 신선함, 샐러드바 12종 제공"
- "고기는 1++ 등급 한우, 입에서 살살 녹는 식감"
- "이유식 의자 4개 비치, 유아용 물티슈 무료 제공"
- "주차 20대 가능, 발렛파킹 무료"

#### 정량적/정성적 정보 구분

AI가 정보를 명확히 인식할 수 있도록 정량적 정보와 정성적 평가를 구분한다:

```markdown
| 구분 | 정보 |
|------|------|
| 정량 정보 | 좌석 수: 50석, 1인 평균 가격: 35,000원, 주차: 20대 |
| 정성 평가 | 분위기: 어두운 조명으로 데이트하기 좋음, 서비스: 친절하고 세심함 |
| 편의성 | 이유식 의자 제공, 유아 동반 가능, 단체석 있음 |
| 품질 평가 | 야채 신선도 우수, 고기 퀄리티 최상급, 밑반찬 정갈함 |
```

#### 검색 쿼리 대응 구조

사용자가 AI에게 물어볼 질문을 예상하고 그에 대한 답변을 명확히 작성한다:

- "○○ 지역 주차 가능한 식당" → 주차 정보를 명시
- "아이 데리고 가기 좋은 ○○ 식당" → 이유식 의자, 키즈 메뉴 등 명시
- "○○에서 분위기 좋은 데이트 장소" → 조명, 테이블 간격, 소음 수준 명시
- "○○ 지역 채식주의자를 위한 식당" → 채식 메뉴, 비건 옵션 명시

### 6단계: FAQ 섹션 추가

- 글 하단에 FAQ 섹션을 추가한다.
- 최소 3개 이상 문항을 넣고, 각 답변은 구체적으로 작성한다.
- FAQ 질문에는 핵심/변형 키워드를 자연스럽게 분산한다.

#### 사용자 궁금증 기반 Q&A 작성

FAQ는 사용자가 실제로 궁금해할 내용을 Q&A 형식으로 작성한다.

**질문 작성 원칙:**
- 실제 검색 쿼리 형태로 작성한다.
- "어떻게", "왜", "무엇을", "어디서" 등 의문사를 활용한다.
- 구체적이고 명확한 질문을 만든다.

**답변 작성 원칙:**
- 질문에 대한 직접적인 답변부터 시작한다.
- 추가 맥락이나 팁을 덧붙인다.
- 가능하면 구체적인 수치나 예시를 포함한다.

예시:

```markdown
### Q. ○○ 레스토랑은 주차가 가능한가요?
네, 20대 규모의 전용 주차장이 있으며 발렛파킹 서비스를 무료로 제공합니다. 주말 저녁에는 만석일 수 있으니 인근 공영주차장(도보 3분)을 이용하셔도 좋습니다.

### Q. 아이를 데리고 가도 괜찮을까요?
네, 이유식 의자 4개가 비치되어 있고 유아용 물티슈와 앞치마도 무료로 제공합니다. 다만 저녁 8시 이후에는 데이트 분위기가 강해 오후 6시 이전 방문을 추천드립니다.

### Q. ○○ 레스토랑의 대표 메뉴는 무엇인가요?
1++ 한우 등심 스테이크(45,000원)가 시그니처 메뉴입니다. 고기가 입에서 살살 녹으며, 12종 샐러드바와 수프, 빵이 무제한 제공됩니다.
```

### 7단계: 작성자 전문성 표기

- 작성자 이름만 적지 말고 전문성 근거를 함께 명시한다.
- 예: 실무 연차, 담당 분야, 자격/프로젝트 경험
- 본문 또는 작성자 박스에서 독자가 확인 가능해야 한다.

### 8단계: 주제 집중 전략 유지

- 각 섹션이 중심 주제와 직접 연결되도록 유지한다.
- 부수 주제가 길어지면 삭제하거나 내부 링크로 분리한다.
- "긴 상세 콘텐츠"를 쓰되, 중심 질문에 대한 답을 벗어나지 않는다.

## 키워드 운용 규칙

- 핵심 키워드는 제목, 도입부, 본문 핵심 소제목, FAQ에 고르게 배치한다.
- 변형 키워드는 동의어/질문형/롱테일 표현으로 분산한다.
- 같은 키워드를 인접 문장에 과도 반복하지 않는다.
- 문장 자연스러움을 해치면 반복보다 문맥 일치를 우선한다.

## 최신성 및 내부 링크 규칙

- 본문 상단 또는 하단에 최신 업데이트 날짜를 표시한다.
- 변경 가능성이 큰 정보(버전, 정책, 가격, 절차)는 업데이트 근거를 함께 명시한다.
- 내부 링크는 관련 주제 문서로 연결하고, 앵커 텍스트를 구체적으로 작성한다.
- 내부 링크는 독자의 다음 행동 경로를 만드는 용도로 배치한다.

## JSON-LD 구조화 데이터 규칙

기본 타입은 `BlogPosting + FAQPage`를 함께 사용한다.

### 필수 필드

- `@context`
- `@type`
- `headline`
- `description`
- `datePublished`
- `dateModified`
- `author`
- `mainEntityOfPage`
- `inLanguage`
- FAQ의 `mainEntity`(Question/Answer 배열)

### 삽입 위치

- 포스트 렌더링 시 `<script type="application/ld+json">`로 삽입한다.
- 본문과 FAQ 내용이 바뀌면 JSON-LD도 즉시 동기화한다.

## Humanizer 조건부 검수 절차

포스트 본문 작성이 끝나면 아래 절차를 수행한다.

### 1차: AI 흔적 탐지

아래 징후가 보이면 AI 흔적 "감지"로 판단한다.

- 쉼표/접속사 과다로 리듬이 단조로운 문장
- 유사한 문장 구조가 반복되는 패턴
- 과하게 일반화된 표현, 번역투 표현
- 지나치게 균질한 문체로 인간적인 변주가 부족한 경우

### 2차: 감지 시 Humanizer 적용

- AI 흔적이 감지된 경우에만 `humanizer` 스킬을 적용한다.
- 문장 자연화는 수행하되, 사실/의미/의도를 변경하지 않는다.

### 3차: 의미 보존 검증

Humanizer 적용 후 아래 항목이 유지되는지 검증한다.

- 사실 정보(수치, 날짜, 버전, 절차)
- 링크 URL 및 앵커 의도
- 핵심 키워드/변형 키워드의 배치 의도
- 작성자 전문성 정보

### 4차: 미감지 시 통과

- AI 흔적이 뚜렷하지 않으면 Humanizer 단계는 `수정 없음`으로 종료한다.

### 변경 로그 규칙

- Humanizer 적용 시, 바뀐 문장 유형을 3~5개 항목으로 요약한다.
- 요약에는 "왜 바꿨는지"를 짧게 포함한다.

## 테스트 케이스 및 검증 시나리오

1. 제목이 평서문이면 질문형 제목 교정 지시가 출력되어야 한다.
2. TL;DR 섹션이 없으면 3-5줄 요약 추가 지시가 출력되어야 한다.
3. 도입 3줄 내 핵심 정의가 없으면 정의 문장 추가 지시가 출력되어야 한다.
4. 본문이 장문 단락 위주면 리스트/테이블 전환 지시가 출력되어야 한다.
5. 장소/제품 리뷰 시 속성 정보 테이블이 없으면 추가 지시가 출력되어야 한다.
6. 모호한 표현("좋다", "맛있다")이 있으면 구체적 속성 표현으로 전환 지시가 출력되어야 한다.
7. FAQ가 없으면 최소 3문항 FAQ 생성 지시가 출력되어야 한다.
8. FAQ 질문이 검색 쿼리 형태가 아니면 교정 지시가 출력되어야 한다.
9. JSON-LD가 없으면 `BlogPosting + FAQPage` 스니펫이 생성되어야 한다.
10. AI 흔적이 감지되면 Humanizer 적용과 변경 요약이 포함되어야 한다.
11. AI 흔적이 미감지면 Humanizer 결과가 `수정 없음`으로 표시되어야 한다.
12. 내부 링크/최신 업데이트 표기가 누락되면 실패로 판정해야 한다.

## 실패/예외 처리

- 필수 입력이 부족하면 누락 항목을 먼저 요청한다.
- 근거 없는 사실, 수치, 출처를 임의 생성하지 않는다.
- 정확성에 확신이 없으면 "이 내용은 정확하지 않을 수 있습니다."를 명시한다.
- 지식 범위를 벗어나는 정보는 "이 정보는 제 지식 범위를 벗어납니다."로 처리한다.
